Geometry

Height 58vh (build range 55–60). Top radius 16px. The sheet and the page share #0d0b09 — the edge is carried by a hairline rgba(255,255,255,0.09) plus 0 -16px 48px rgba(0,0,0,0.55), never by a surface-color change.

Drag handle

36 × 4px, radius 2, rgba(240,234,216,0.18). The touch target is the full-width 28px strip around it. Drag up → expanded; drag down → dismiss. Rendered on touch devices only (P3).

Backdrop — R1

No scrim at partial. Seeing the plan is the point of this state; dimming it would defeat it.

Background — R2

The plan above the sheet stays live: it scrolls and its cards respond to taps. The sheet is non-modal at partial. BottomNav sits behind the sheet's lower half and is unreachable until dismissal — tab switching mid-conversation would silently change the context a turn carries.

Keyboard — R3

The input bar rides the visual viewport and pins above the keyboard; the sheet's top edge does not move. The thread shrinks and scrolls — the sheet never auto-expands (P2). The user can drag up if they want more room.

Streaming — P2

Long replies scroll inside the thread at the current height. Content growth never changes sheet height; auto-scroll-to-bottom behavior is preserved from the current panel.

Header

Title left in DM Mono 11px / 0.28em, close × right (32px target). The streaming Stop button and owner-only clear keep their current slot at the right edge (P8).

Reaching it

Drag up from partial (touch). The header also gains a collapse control ⌄ at expanded — drag remains primary on mobile, the control is the explicit way back down.

Transition

Spring easing via the Motion library — the idiom design spec §8.7 already names for the co-pilot. Top radius interpolates 16px → 0 across the drag. Respects prefers-reduced-motion.

Modality

Expanded is modal — the plan is fully covered on mobile, so the question of background interaction resolves itself. Dismissal returns to partial (drag down or ⌄), not straight to closed; a second drag-down closes.

P8 — nothing re-learned

Proposal cards, apply / reject, loss warnings, status copy (Applied · Dismissed · Outdated · Can't Apply), attribution labels, owner-only clear: identical components, re-parented into the sheet. The proposal card shown is the shipping MutationPreview pixel-for-pixel.

Derivation — P6

A pure frontend util (utils/copilotSeeds.js) reads the trip detail already in TripPage. Three slots, in priority order: active-day density (stop count / timing gaps on the day being viewed), active-day appetite (a food-shaped gap in that day's stop mix), and the next upcoming booking (here, Wednesday's Hongqiao → Hangzhou East train). No LLM call — copy is templated from real names.

Tap behavior

A tap sends the prompt as an ordinary message carrying the current context — it renders as a user bubble with its chip, exactly like a typed turn. Nothing is pre-filled into the input.

Type

Prompts in Cormorant Garamond 16.5px — they are trip narrative, not UI chrome. Eyebrow in DM Mono gold is this component's single gold accent; arrows stay cream-mute.

Degenerate trips

A trip with nothing to derive from (no stops, no bookings) gets one neutral prompt — "What's worth knowing about this trip?" — never lorem, never a blank panel.

Screen 04 — Component

The context chip

Whatever context a turn carries is visible on that message — never silently injected (P5). One idiom covers every source: FAB, stop entry point, Discovery, seed prompts, history.

Plan · Day 3 · Hangzhou Today · Day 2 Map · Day 3 Discovery · Hupao Spring
GrammarSurface first, then the most specific object: TAB [· DAY n · CITY] [· STOP]. The server resolves ids to names; the chip shows names only.
TypeDM Mono 9px, uppercase, 0.16em tracking — the existing badge language.
ColorText rgba(201,168,76,0.9), border 1px rgba(201,168,76,0.35), transparent fill, radius 4. The chip is the one gold on a message row — user bubbles stay cream-on-surface.
PlacementRight-aligned above the user bubble, 5px gap. In multi-traveller threads the author label sits on the same line, left of the chip.
HistoryPersisted with the message (context_json, migration 029) — a refreshed thread renders chips identically to a live one.
AbsentA turn with no context renders exactly as today: no chip, no placeholder.

Why a chip, not prose

The user should never wonder what the co-pilot was told about where they were. A quiet mono chip states it in the product's own label language without inflating the bubble.

Server side — for the record

Context is validated (known tab, ids belong to the trip), then composed into the user turn as a bracketed line — never the cached system prompt (Plan 13 fact 6). Invalid context is dropped with a log, never a failed turn.

Pointer-native, no gesture cosplay

Same sheet component, different affordances (P3): no drag handle, an explicit expand control, hover states, Escape to close. The sheet caps at 640px and centers — a full-width bar across a wide viewport would read as a drawer, not a companion.

Controls

⌃ expands (becomes ⌄ at expanded), × closes, Esc closes. 32px targets, hover fill rgba(255,255,255,0.06), radius 6. No drag handle rendered, no hover-triggered motion.

Geometry

Width min(640px, 100% − 48px), centered. Partial 56% of viewport height; expanded 92% — the sheet never touches the desktop's top edge, so it always reads as a layer on the plan.

Scrim — R1, desktop

None at partial. At expanded the app remains visible at the sides, so a scrim earns its keep here: rgba(13,11,9,0.55) behind the sheet, click-to-collapse.

For sign-off

Decisions this approval fixes

Four recommendations and the v1 entry-point set. Approval binds Waves 2–4; anything excluded stays excluded unless added here.

R1

No scrim at partial; scrim only where the app stays visible at expanded.

Partial height exists to keep the plan readable — dimming it would defeat the feature. Mobile-expanded covers the viewport (scrim moot); desktop-expanded dims the exposed sides at rgba(13,11,9,0.55).

R2

The sheet is non-modal at partial: the plan scrolls and responds behind it.

Reading the day while talking about it is the core loop. BottomNav stays covered while the sheet is open — switching tabs mid-conversation would silently change what a turn's context means.

R3

Keyboard never resizes the sheet — the thread absorbs it.

The input bar pins above the keyboard via the visual viewport; the sheet's top edge holds. This is P2 (no uninitiated motion) applied to the keyboard, the case most sheet implementations get wrong.

R4

The sheet edge is structural, not chromatic: hairline + shadow + 16px radius.

Sheet and page share the same ink. A lighter sheet surface would break the palette; a gold edge would spend the accent on furniture. Depth does the work instead.

Entry-point set, v1

  • IN · EP-1Expanded stop card action row — Ask co-pilot → (non-transit stops), carrying tab · day · stop.
  • IN · EP-2Discovery suggestion card action row — Ask co-pilot, carrying the suggestion's name.
  • OUTDay headers — the context-aware FAB already knows the active day; a header control would be new persistent chrome.
  • OUTCollapsed stop cards, transit rows, map pins — nothing new renders outside an already-opened action surface.
  • OUTLogistics booking cards ("what does this booking affect?") — deliberately deferred; add only via a future sign-off.
